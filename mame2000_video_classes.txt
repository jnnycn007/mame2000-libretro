mame2000 per-game video class list
======================================================================

WHAT THE CLASSES MEAN
----------------------------------------------------------------------
The libretro build of mame2000 always delivers RGB565 frames to the
frontend.  Every game goes through one of three blit functions in
src/libretro/blit.c (dirty0/dirty1 variants of each are identical in
per-pixel cost; the dirty1 forms skip blocks the driver hasn't marked
modified).  Which blit runs for a given game is set at osd_create_display
time from the game's preferred depth and palette mode:

  COLOR8_LUT  - !GAME_REQUIRES_16BIT
                -> blitscreen_dirty?_color8
                -> per pixel: gp2x_palette[index]    (8-bit pen -> RGB565)

  PAL16_LUT   -  GAME_REQUIRES_16BIT && VIDEO_MODIFIES_PALETTE
                -> blitscreen_dirty?_palettized16
                -> per pixel: palette_16bit_lookup[pen]
                              (16-bit pen -> RGB565)

  DIRECT16    -  GAME_REQUIRES_16BIT && !VIDEO_MODIFIES_PALETTE
                -> blitscreen_dirty?_color16 used as a fallback only.
                The blit is a pure row-by-row memcpy; the fast path in
                video.c skips it entirely and hands the MAME scrbitmap
                pointer straight to video_cb, giving real zero-copy
                delivery when the visible area exactly matches the
                libretro frame size (the typical case).

  VECTOR      -  VIDEO_TYPE_VECTOR
                -> dynamically-sized bitmap from the vector renderer
                   that lands in the color16 blit downstream.
                Per pixel cost equal to DIRECT16.

WHAT THE BLIT SHARES WITH AUDIO RATE-OVERRIDE
----------------------------------------------------------------------
The blit path is the same regardless of whether the audio rate has
been overridden -- the audio rate-override patch is independent and
both improvements can stack.

The software-framebuffer fast path added in retro_run lets the active
blit write directly into the frontend's framebuffer when granted,
removing one full-frame copy in the libretro pipeline.  COLOR8_LUT
and PAL16_LUT save the LUT result straight to the frontend; VECTOR
saves the bitmap straight to the frontend.

DIRECT16 has an additional bitmap-direct fast path: when the bitmap's
visible area equals the libretro frame size and is at offset (0,0),
video.c skips the blit and retro_run hands the scrbitmap pointer to
video_cb with the bitmap's row stride as pitch.  This bypasses the
intermediate gp2x_screen15 entirely and is unconditional zero-copy --
the only DIRECT16 frames that still go through the (now mask-less)
blit are user-panned frames and frames whose visible area is a strict
subset of the bitmap.

OPTIONS THAT AFFECT DEPTH
----------------------------------------------------------------------
options.color_depth is 0 in the libretro build (config.c is unwired),
so a game ends up at 16bpp only when its GAME() macro sets the
GAME_REQUIRES_16BIT flag via the ROT*_16BIT shortcuts.  This means
several games whose machine driver sets VIDEO_MODIFIES_PALETTE but
whose monitor flag is bare ROT* (e.g. some CPS1 titles, neogeo) end
up in COLOR8_LUT here even though their colour count exceeds 256;
those games still run because mame2000's palette-shrinking machinery
fits the visible subset into the available pens.

ORIENTATION
----------------------------------------------------------------------
Each line lists the game's monitor orientation (ROT0/ROT90/ROT180/
ROT270) as declared in its GAME() macro.  Orientation does not change
the blit's per-pixel cost; MAME rotates the bitmap before the blit
runs.  The orientation column tells you which way the frontend will
display the game without any rotation applied on its end.

DIFFERENCES FROM MAME2003
----------------------------------------------------------------------
mame2003 has a 'video formats' catalogue describing whether each
game's frame can be zero-copied to the frontend.  mame2000's blit
pipeline always converts (LUT or mask), so true zero-copy isn't
applicable; the closest analogue is the software-framebuffer path
which makes the conversion a single pass into frontend memory.
